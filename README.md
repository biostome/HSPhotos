# HSPhotos 产品定义书

本文档**定义 HSPhotos 是一款什么样的应用**：面向谁、解决什么问题、提供哪些能力、与系统「照片」如何分工、哪些事情明确不做。**第五节「核心功能」**逐条写清用户侧行为、产品应达成与协作注意点，供后续 AI 与工程直接对照。产品、设计、研发以及**协助本工程的 AI** 均应以此作为讨论需求与方案时的**共同基准**。

**与实现的关系**：真实行为以当前可交付产品为准；若实现与本文不一致，应**优先修正产品或修正实现**之一，并**回写本文**，避免文档沦为空话。

## 二、产品定位

### 2.1 一句话定义

**HSPhotos 是运行在 iPhone 上的本地相册客户端**：在系统授权范围内读取与修改用户的**系统照片库**，以「图库总览 + 相册管理 + 单相簿深度整理」三层结构，帮助用户**浏览、排序、整理、标注与编排**照片与视频。

### 2.2 解决的核心问题

- **与系统顺序对齐**：相簿内「自定义排列」应以系统照片库为准，避免应用内出现一套与系统「照片」长期打架的顺序。
- **整理效率**：在相簿内提供多选、范围选择、锚点式批量重排并写回系统等能力，服务「我要按自己的逻辑排一版相册」的场景。
- **表达结构**：在相簿内提供多级编号、折叠、首图与段落等能力，服务「我要把相册当成有层级的大纲来用」的场景。
- **本地标注**：通过应用内标签与筛选，在**不依赖系统关键词体系**的前提下，为照片建立可检索的私有分类。

### 2.3 与系统「照片」的关系

| 关系 | 说明 |
|------|------|
| **延伸** | 在系统能力之上，强化**顺序编排、层级化展示、标签筛选**等整理向能力。 |
| **不替代** | 不追求替代系统「照片」的全部能力（尤其是专业修图、完整 RAW 工作流、回忆与推荐等）。 |
| **数据归属** | 照片与视频实体仍在用户系统照片库中；本应用主要增加**偏好与元数据层**（排序方式记忆、编号、折叠、标签等）。 |

## 三、目标用户与典型情境

### 3.1 目标用户画像（可重叠）

- **整理型用户**：大量活动/项目照片，需要按讲述顺序排进相簿，并可能导出或给他人看。
- **大纲型用户**：希望用「1、1.1、2」式层级在相册里表达结构（汇报、教案、展览顺序等）。
- **检索型用户**：习惯用自建标签快速缩小范围，而不是只依赖拍摄时间滚动。

### 3.2 典型情境（叙事）

1. **快速总览**：打开图库，按时间或「最近加入」扫一遍全库，不做重度编排。  
2. **管理相簿壳**：在「相册」里找文件夹、新建相簿、把照片加进目标相簿。  
3. **深度编排某一相簿**：进入相簿网格，切换排序、多选、用锚点把一批照片挪到故事线某处，确认后顺序写回系统。  
4. **结构化浏览**：打开层级编号与折叠，把长篇相册收成几段；必要时标首图、看段落关系。  
5. **按标签收敛**：给一批照片打标签，用筛选或搜索栏快速定位子集。

## 四、产品形态与信息架构

用户心智中应存在**三个层级**，职责不可混淆：

| 层级 | 名称（产品用语） | 在用户心智中的角色 | 能力重心 |
|------|------------------|----------------------|----------|
| **L1** | 图库 | 「我全部照片在哪」 | 总览、轻量排序；**不做**与单相簿深度绑定的那套层级编号工作流。 |
| **L2** | 相册 | 「我的相簿和文件夹怎么摆」 | 列表与组织；列表自己的排序**只决定相簿条目怎么排**，**不等价**于某一相簿内部照片顺序。 |
| **L3** | 相簿内网格 | 「这个相簿里照片怎么排、怎么标」 | 网格、多选、排序切换、写回顺序、层级编号与折叠、标签筛选等**重操作**。 |

**相簿内排序偏好**按相簿独立记忆：用户从 A 相簿切到 B 相簿，不应默认「B 沿用 A 的排序方式」，除非产品明确改为全局偏好。

## 五、核心功能（协作必读）

以下列出 HSPhotos **已实现或可交付的核心能力**。后续协作（含 AI）在**新增需求、改动行为、排查缺陷或评估影响**时，应默认这些能力**已存在且需被尊重**；若要做删减或替换，须显式走产品决策并回写本文。

**操作级细节**（控件在屏幕上的位置、点击顺序、灰态与成功/失败表现）见 **第十二节**；第五节保留「是什么 + 协作注意」的浓缩版。

### 5.1 图库（全库总览）

| 维度 | 说明 |
|------|------|
| **用户做什么** | 查看设备上「全部照片」意义上的总览；在有限几种排序方式间切换以快速浏览。 |
| **产品应达成** | 总览体验尽量贴近用户对系统「照片」同类视图的直觉；**不提供**与「单相簿内多级编号、层级折叠」绑定的同一套重度能力。 |
| **协作时注意** | 不要把相簿内那套编号工作流默认搬到图库；图库与相簿的菜单与能力集**有意区分**。 |

### 5.2 相册列表（相簿与文件夹）

| 维度 | 说明 |
|------|------|
| **用户做什么** | 浏览用户创建的相簿与文件夹层级；进入某一相簿；新建、重命名、删除、多选、向相簿添加照片等组织动作。 |
| **产品应达成** | 列表层级的排序（按名称、按修改时间等）**只影响列表上条目的排列**，不改变用户尚未打开的那个相簿**内部**照片顺序。 |
| **协作时注意** | 讨论「排序」时必须指明是**列表排序**还是**相簿内排序**，避免混用同一套规则或同一套记忆键。 |

### 5.3 相簿内网格（浏览与展示）

| 维度 | 说明 |
|------|------|
| **用户做什么** | 在某一相簿中以网格查看缩略图、滚动浏览；在开启层级与折叠时，看到的是**折叠后的可见子集**。 |
| **产品应达成** | 与「第几行、第几张」相关的交互以**当前屏幕上可见顺序**为准；大库下保持可接受的滚动与进入体验。 |
| **协作时注意** | 任何依赖下标或顺序的功能，须区分「全相簿成员顺序」与「过滤后子集」与「折叠后可见行」三层语义，避免默认三者恒等。 |

### 5.4 排序与自定义顺序

| 维度 | 说明 |
|------|------|
| **用户做什么** | 在相簿内（及图库在允许范围内）切换多种时间类排序；选择「自定义顺序」以按系统相簿内用户排列浏览。 |
| **产品应达成** | 界面所宣称的当前排序方式，必须与**从系统照片库取列表时实际采用的规则**一致；自定义顺序**不以应用内第二套持久顺序表**替代系统。 |
| **协作时注意** | 凡导致「界面已是自定义、拉取仍按日期」的状态不同步，均属产品缺陷；改排序相关逻辑时须整体检查「界面偏好、取数规则、网格展示」是否仍对齐。 |

**顺序语义（须区分的三层）**

1. **系统相簿成员顺序**：用户在系统「照片」中看到的该相簿内照片顺序；**自定义排序**模式下取数**不再附加**日期类排序描述符，顺序与系统该相簿一致。  
2. **应用内当前成员数组顺序**：进入相簿后加载到内存的列表顺序；批量写回、粘贴插入等逻辑均以此与系统双向对齐。  
3. **网格上「可见行」顺序**（仅当同时启用**按自定义排序**且该页**支持层级**时）：在 2 的基础上再经**层级折叠**过滤（见 5.7），用于滚动与选择命中；**不得**与 1、2 混称为「系统顺序」。

**各排序选项（相簿侧，与 `PhotoSortPreference` 一致）**

| 选项（用户可见文案） | 取数规则概要 |
|----------------------|--------------|
| **按自定义排序** | 不施加额外 `NSSortDescriptor`，顺序与系统该相簿成员顺序一致。 |
| **按拍摄日期排序** | `creationDate` 升序，`modificationDate` 升序为辅。 |
| **按修改时间排序** | `modificationDate` 降序，`creationDate` 降序为辅。 |
| **按最近加入时间排序** | `modificationDate` 降序，`creationDate` 降序为辅。 |
| **按最旧的排最前排序** | `creationDate` 升序，`modificationDate` 升序为辅。 |
| **按最新的排最前排序** | `creationDate` **降序**。 |

**偏好记忆**：相簿内当前排序方式按**相簿**单独持久化（与「相册」Tab 上**相册条目**的排序记忆不是同一套键，见第四节）。

### 5.5 选择与批量重排并写回系统

| 维度 | 说明 |
|------|------|
| **用户做什么** | 单选、多选、范围选择；设锚点；将选中照片按意图重排后，将**整相簿新顺序**写回系统照片库。 |
| **产品应达成** | 写回成功后，系统「照片」及其他应用打开该相簿应看到一致顺序；失败时明确提示，避免长期「假成功」。 |
| **协作时注意** | 写回通常要求全相簿成员集合一致；批量重排一般至少两张选中；锚点缺失或选中不足时应有清晰错误反馈。 |

**选择模式**

- **非选择**：点格进入大图等默认行为。  
- **多选**：逐张点按切换勾选；内部维护**选中顺序**（对后续批量重排有意义）。  
- **范围选择**：打开导航栏上的范围开关后，在网格上以滑动手势在**连续索引区间**内批量勾选或取消；索引相对于**当前网格所绑定的 `assets` 序列**（见下「与标签筛选」）。

**锚点（排序基准）**

- 用于表达「**这一批要移动的照片，整体插在锚点之后**」。  
- **锚点不必在选中集内**；若已设锚点，排序时以锚点为基准，**其余选中项按当前选中顺序**依次排在锚点之后。  
- **未设锚点**时，以**选中顺序中的第一张**同时承担「基准」与「选中首项」：其余选中项按选中顺序排在该张之后（基准张本身不移动）。  
- 取消选中「当前作为锚点」的那一张、执行**全选**、或退出选择并清空，会**清除锚点**。

**批量重排（内存算法概要）**

1. 若选中张数 `< 2`，不允许执行排序（用户可见失败说明，如「至少需要选择两张照片…」）。  
2. 从当前网格持有的 `assets` 序列中，暂时**移除**「待移动集合」（= 全部选中项；若锚点在选中集内则去掉锚点这一张，使锚点留在原位作墙）。  
3. 在剩余序列中定位**基准照片**（锚点优先，否则为选中顺序首张），将「待移动集合」按**选中顺序**整体插入到 **基准索引 + 1** 之后。  
4. 得到完整新序列后，调用系统相册变更接口**整表替换**该相簿内成员顺序；服务层校验：新序列与系统当前成员集合**数量相同、元素集合相同**，否则拒绝写回（防止子集或错集误同步）。

**与标签筛选、层级折叠**

- **标签筛选开启**时，网格常展示相簿的**子集**；若排序算法仅基于该子集生成新序列，则长度往往小于系统相簿成员总数，**写回校验会失败**。产品侧应引导：整相簿重排宜在**未筛选的全量视图**下完成；若未来支持筛选态重排，须单独定义「是否先与全量合并」并配套提示。  
- **层级折叠**改变的是**展示用可见序列**与多选命中范围；批量写回仍以**全量成员顺序**为权威（与上述校验一致）。

### 5.6 剪贴板与相簿间资产操作

| 维度 | 说明 |
|------|------|
| **用户做什么** | 复制选中项到剪贴板；在相簿内指定位置粘贴；创建副本；移动到其他相簿；按系统规则从相簿移除或删除；向相簿添加照片（含从系统选择器选取）。 |
| **产品应达成** | 操作后果符合 iOS 对「全部照片」智能视图与用户自建相簿的差异（删除、撤销可能性等）；耗时操作有进度或完成反馈。 |
| **协作时注意** | 粘贴、插入位置若与「标签过滤后的可见列表」混用，易产生心理模型冲突；需明确产品语义或消歧，不宜静默选错参照系。 |

### 5.7 相簿内层级编号、折叠与可见性

| 维度 | 说明 |
|------|------|
| **用户做什么** | 为照片设置多级编号；折叠/展开子树；在策略允许下控制「间隙」等展示规则。 |
| **产品应达成** | 编号与折叠为应用侧元数据，随成员增删与顺序变化做清理与校正，避免出现无法理解的状态。 |
| **协作时注意** | 该能力仅在相簿内等已定义场景提供，图库总览不提供同一套；改动规则时须考虑对可见行数、角标与选择态的连锁影响。 |

**层级（level）与编号**

- 每张图可为 **无层级（0）** 或 **有层级（≥1）**；仅后者参与大纲式编号。  
- **编号字符串**（如 `1`、`2.1`、`3.2.1`）由**当前全量成员顺序 + 各 asset 的 level** 经栈式规则**计算**得出，用户不可直接编辑数字串；顺序变化时同级计数器会重算。  
- **层级合法性**：若某张存储的 level 相对前一张有编号项过大（超过「前序 level + 1」），实现会按与编号计算一致的规则**回落**到合法深度，避免出现无法渲染的大纲洞。

**折叠（collapse）与可见行**

- 用户对某一**存在更深子项的层级节点**可切换「折叠/展开」；折叠时，**level 比该节点更深的后续成员**在网格中隐藏，直到遇到**同级或更浅**的有编号节点，或遇到下列「间隙」规则边界。  
- **无编号间隙策略**（在叠加信息「详细设置 → 层级折叠 → 无编号间隙」中二选一）：  
  - **断开**：中间 **level = 0** 的无编号照片会**打断**折叠链，折叠不跨过该图继续影响更远处的项。  
  - **含间隙**：在「后面第一个有编号项仍比折叠根更深」时，其间的无编号照片可**随子树一并隐藏**；若下一个有编号项已回到同级或更浅，则其间无编号图**不**随折叠隐藏（与设置页脚说明一致）。  
- **可见集合**：仅在 **按自定义排序** 且该页 **supportsHierarchyNumbering == true** 时，网格在 `assets` 全量上套用层级折叠得到 `visibleAssets`；其它排序下网格顺序即全量顺序，不套层级折叠。

**持久化与清理**

- level、折叠布尔按相簿存于应用侧存储；进入/离开相簿时加载与保存。  
- 成员从相簿移除时，应清理对应元数据并对剩余 level **再校正**，避免「幽灵编号」或无效折叠键。

### 5.8 首图与段落

| 维度 | 说明 |
|------|------|
| **用户做什么** | 将某些照片标为段落首图，与折叠等配合，把长卷相册分成多段阅读。 |
| **产品应达成** | 段落语义与「系统相簿顺序」正交；成员删除或移动后段落仍应在用户侧可理解。 |
| **协作时注意** | 与「仅编号、仅折叠」区分职责，避免同一交互在段落与层级两套语义间含糊。 |

**数据模型（领域语义）**

- **首图列表**：在某一相簿内，用户可标记**一张或多张**「段落首图」；持久化为首图 asset 标识的有序列表。  
- **段落划分**：按**当前相簿内成员顺序**从左到右扫描——每遇到一张首图，即开启**新段落**；该首图之后、下一张首图之前的所有非首图成员，归入该首图所属段落的**跟随内容**。  
- **段落折叠**：每个首图可独立处于「段落展开 / 段落折叠」；**折叠时首图本身始终仍显示**，仅隐藏该段内**跟随**成员；展开时跟随成员重新出现。  
- **尚无首图时**：领域上可把顺序中靠前位置视为「未命名段」的占位，使未标首图前的成员仍落在可解释的段落内（具体占位规则以实现为准）。

**与层级编号的关系**

- **层级折叠**服务「大纲编号树」的扫读；**段落**服务「长卷分成阅读段」的心智；产品上不互相替代。  
- 若同一相簿同时使用两套能力，**屏幕上最终可见行**由实现中过滤器组合顺序决定；测试与验收应以**真实网格**为准。

**与主网格的衔接**

- 首图与段落状态由专用服务维护；与主网格数据源的结合程度以**当前可交付版本**为准。上文定义**领域目标**；若部分入口仍在接入中，应以路线图或变更说明追踪，而非弱化语义定义。

### 5.9 标签、筛选与搜索栏

| 维度 | 说明 |
|------|------|
| **用户做什么** | 为照片添加应用内标签；在筛选面板按「满足任一标签 / 满足全部标签」过滤；在搜索栏按标签名匹配或跳转到指定序号（若产品提供）。 |

| **协作时注意** | 新增与标签、筛选、搜索栏交互的功能时，须声明它作用在「全量」还是「当前可见子集」上，避免下标与顺序混用。 |

### 5.10 缩略图叠加信息

| 维度 | 说明 |
|------|------|
| **用户做什么** | 控制是否在格子上显示选择序号、层级编号、拍摄/修改日期、以及在非自定义排序下是否显示「在系统相簿顺序中的序号」等。 |

## 六、能力版图（产品层补充）

以下按**用户价值**对第五节作补充叙述；第五节为**能力清单与协作要点**，本节为**连贯说明**。

### 6.1 浏览与导航

用户应能顺畅在图库、相册列表、某一相簿网格之间切换；大库下滚动与进入相簿的体验应**可接受**（具体性能指标由工程与体验测试定义，本文不承诺数值）。

### 6.2 排序与顺序语义

用户可在**不同上下文**下选择不同排序语义：按拍摄时间、按加入或修改时间、最旧或最新在前，以及**自定义顺序**（与系统「照片」中该相簿的成员顺序对齐）。图库与相簿的**可选菜单项集合不同**（见 12.6），但原则相同：**当前勾选项即对用户承诺的取数规则**。

**产品承诺**：界面所展示的当前排序方式，应与**从系统照片库取数时采用的规则**一致；否则用户会看到「选了自定义，一刷新又变回按日期」等**不可信**行为——这在产品层面视为**缺陷**，而非「用户理解问题」。

**自定义顺序的本质**：不在应用内长期维护一套与系统冲突的「第二套顺序表」；顺序以系统为准，应用通过写回等操作改变系统后，再与系统保持一致。

**与「可见行」的关系**：一旦用户启用层级折叠（且为自定义排序），屏幕上看到的是**折叠后的子集**，但**系统侧成员顺序**仍以未折叠前的全量线性序为底；讨论「写回」「粘贴到此后方」时，心里要同时有**全量序**与**可见序**两把尺子（详见 5.4、5.7、6.6）。

### 6.3 选择、锚点与批量重排

用户可进行单选、多选、范围选择。多选与范围选不仅产生「集合」，还产生**顺序**：同一组照片若以不同顺序被点选或划选，后续**批量重排**时插入块内部的相对顺序也随之不同——这是刻意保留的表达能力，服务于「先选中的先跟在锚点后面」等细颗粒意图。

**锚点**解决的是「**插到谁后面**」这一维：锚点可以不在选中集内，表示「把我选中的这一串，整体搬到某张参考图之后」；不设锚点时，默认以选中顺序中的**首张**为墙，其余跟随其后。锚点在交互上应可被随时取消或覆盖，避免用户带着陈旧基准误触排序。

**写回**阶段，系统相册 API 要求新顺序与成员集合**严格可对账**（数量与元素一致）。因此：任何只改「子集顺序」而不给出完整新排列的尝试，都应被拦下或引导用户回到全量上下文。写回失败时，产品应**明确反馈**，并避免长期呈现与系统不一致的「假成功」状态。

### 6.4 相簿与资产的常见操作

用户应能完成与整理强相关的操作，包括但不限于：复制到剪贴板、在相簿内指定位置粘贴、创建副本、移动到其他相簿、从相簿移除或按系统规则删除、向相簿添加照片等。

**与系统类型相关**：「全部照片」类视图与用户自建相簿，在**删除是否进最近删除、是否可撤销、是否仅移出相簿**等语义上可能不同——产品应向用户呈现**符合系统规则**的文案与后果，不能假设所有相簿同一种删法。

### 6.5 过滤视图下的一致性

当用户按标签等条件**过滤**网格时，屏幕上所见为**子集**。任何「插入到某张照片之后」的操作，必须定义清楚：参照的是**过滤前顺序**还是**当前可见顺序**，并在心理模型困难时用提示或交互消歧；否则易出现「我点的是屏幕上这一张，结果却插到别处」的挫败感。

### 6.6 层级编号、折叠与可见行

在相簿内，用户可为照片赋予**多级大纲编号**（形如 `1`、`1.2`、`2.3.1`），并通过**折叠**暂时隐藏某一节点之下的深层子项，使长相册在视觉上变短、更像「可展开的大纲文档」。编号与折叠**不改变**系统里照片的物理归属，只改变**本应用内如何展示与如何扫读**——除非用户再执行**写回顺序**等会触及系统相簿顺序的操作。

**可见行**由三层因素共同决定：（1）当前排序是否为自定义且是否启用层级能力；（2）每张的 level 与折叠布尔；（3）「无编号间隙」采用**断开**还是**含间隙**策略。用户手指下的「第 3 行」应理解为**当前可见表中的第 3 行**；与「全相簿第 3 张」或「系统里排序第 3 号」在折叠开启时**未必相同**。因此：搜索栏跳号、范围选择、全选等凡带「索引」的操作，都应在产品说明中写清参照的是**可见表**还是**全量表**（当前实现以网格数据源为准，见 5.5、12.7）。

编号与折叠数据属于**应用自创元数据**：当相簿成员增删或顺序大变时，产品应通过规则或校正避免「幽灵编号」「指向已删照片的折叠态」等不可理解状态；并在角标与长按菜单中让用户能**看出**哪张是折叠根、哪段子树被收起。

### 6.7 首图与段落

**首图**不是「封面图」或「精选」的社交语义，而是**阅读分段**的锚：用户在一卷连续照片里标出若干首图，相当于在书页里插入「章标题」。**段落**即「某首图 + 其后直到下一首图之前的所有跟随照片」。段落可整体**折叠阅读**（首图仍在，跟随暂时隐藏），用于把汇报稿、旅行长卷、教案相册收成「先扫章再展开细节」的节奏。

该能力与「系统相簿顺序」**正交**：重排照片不会改变「谁是首图」的标记，除非用户主动改标记或删除成员。删成员时，应清理无效首图标识，并避免段落结构出现「首图指向已不存在的照片」等断层。与**层级编号**并存时，二者解决不同问题——一个是**大纲深度**，一个是**阅读章界**——产品叙事上应分开讲清，避免同一动词在不同菜单里承载两种语义。

### 6.8 标签、筛选与搜索

标签由应用本地维护，用于给照片打标、按「满足任一 / 满足全部」等规则筛选，以及通过搜索栏按名称匹配标签或跳转到指定序号（若产品提供该能力）。标签**不是**系统相册自带的「关键词」体系。

### 6.9 缩略图上的信息层

用户可控制是否在格子上叠加：选择序号、层级编号、日期信息、以及在非自定义排序下是否展示「在系统相簿顺序中的序号」等。开关变更后，展示应**及时一致**。

### 6.10 撤销与重做

对一批相册相关操作提供**应用内**撤销与重做，降低误操作成本。产品不承诺「任何操作都可无损撤销到任意历史点」：当系统本身不可逆时，应向用户诚实呈现。

## 七、体验与设计原则（产品层）

| 原则 | 含义 |
|------|------|
| **可信** | 界面状态与系统照片库真实状态可对账；排序、成员、权限三类信息不互相打架。 |

| **结构可读** | 层级与折叠服务于「扫一眼能懂结构」，避免为炫技增加认知负担。 |

## 十、关键术语（用户语言）

| 术语 | 含义 |
|------|------|
| **图库** | 全库总览入口，不等价于「某一个相簿」。 |
| **相册（列表）** | 相簿与文件夹的目录；列表排序 ≠ 相簿内照片顺序。 |

| **标签** | 应用内自建分类，非系统关键词。 |

| **按自定义顺序排序** | 恢复为系统侧**自定义顺序**语义下的列表顺序（与相簿内「自定义排序」不是同一套记忆键）。 |

#### 12.3.4 列表上的多选

| 步骤 | 说明 |
|------|------|
| **进入** | 仅在**列表视图**下点 **「选择」**；隐藏系统返回按钮；左侧出现 **「完成」**；标题为 **「选择项目」**。 |

| **展开/收起（多选语义）** | 若所选文件夹**已全部展开**，则本次为**收起所选**；否则为**展开所选**（仅作用于选中且可展开的文件夹）。 |

| 操作 | 结果 |
|------|------|

| **导航栏「展开全部 / 收起全部」** | 在**非多选**、**列表视图**、存在可展开文件夹时可用；一次展开或收起**当前列表数据中的全部**可展开文件夹。 |

在**未进入「选择」模式**时，对某一相簿或文件夹行**长按**可调起菜单（多选模式下不提供）。

| 项 | 出现条件 | 结果概要 |
|----|----------|----------|
| **添加照片** | 目标为**相簿**（非文件夹） | 调起系统 **PHPicker**，将所选图片加入该相簿（需权限）。 |

#### 12.3.7 「选择相簿」流程（选择器模式）

用于从其他流程（如添加到相簿）**模态**选择目标相簿时，界面仍为相册列表，但导航栏语义收窄：

| 场景 | 左侧 | 右侧 | 标题 |
|------|------|------|------|
| **根级选择器**（未处于某一文件夹内） | **取消**（dismiss 模态栈） | 无 | **「选择相簿」** |
| **已进入某一文件夹** | 系统**返回**（回到上一层列表） | 无 | 一般为该文件夹标题 |

### 12.6 排序 — 图库与相簿的差异

| 场景 | 入口 | 菜单项（文案） | 结果概要 |
|------|------|----------------|----------|
| **图库** | 导航栏上的排序按钮 | **按最近添加排序**、**按拍摄日期排序** | 全库列表按对应时间语义重排；「最近添加」侧更贴近系统默认顺序的设计意图。 |
| **相簿** | 底部「排序方式」浮动按钮（非选择模式） | 当前实现常见为三项：**按自定义排序**、**按最新的排最前排序**、**按最旧的排最前排序** | 相簿内列表按选项重排；**自定义顺序**表示与系统该相簿成员顺序一致，而非应用私有序。 |

**与第五节对照**：相簿侧完整枚举还包含「按拍摄日期 / 按修改时间 / 按最近加入时间」等变体（见 5.4 表），用于取数与偏好记忆；**界面入口**可能只暴露其中子集，以当前版本为准。无论入口多寡，**勾选态与 `PHFetchOptions` 描述符必须一致**。

**条件**：切换排序后，界面所选与列表拉取规则必须一致（见 5.4 / 6.2）；否则视为产品缺陷。

### 12.7 选择模式、全选与范围选择

| 项目 | 说明 |
|------|------|
| **进入选择** | 点导航栏 **「选择」**。 |
| **退出选择** | 点 **「取消」**（叉或文案为取消的按钮）；通常会清空选中与锚点。 |
| **多选** | 在选择模式下逐张点缩略图，出现选择角标；可再次点击取消选中。**点选顺序会进入「选中顺序」**，影响 12.15 批量排序时块内相对顺序。 |
| **全选 / 取消全选** | 选择模式下点导航栏左侧按钮；作用对象为**当前网格数据源中的全部项**（若存在标签筛选，则为**过滤后的子集**，见 5.5、12.13）。 |
| **范围选择** | 选择模式下打开 **范围选择** 开关后，在网格上滑动以划选连续区域（具体手势以实际为准）；区间内勾选顺序与滑动方向相关。 |
| **设锚点（用于批量排序）** | 见 12.10 长按菜单中的「设为锚点」；锚点用于决定一批选中照片插入的基准位置。 |
| **条件** | 未进入选择模式时，依赖「当前已选中」的菜单项（见 12.8）呈灰色或不可点。 |

#### 12.7.1 锚点与选中顺序（与 12.15 配套）

| 行为 | 说明 |
|------|------|
| **设为锚点** | 当前格成为批量排序的**插入参考**；**不要求**该格处于选中状态。 |
| **取消锚点** | 清除参考；此后批量排序默认以**选中顺序首张**为墙。 |
| **取消选中「当前锚点格」** | 锚点一并清除，避免无意识的旧基准。 |
| **全选** | 会清除锚点（与「整张表重选」心智一致）。 |
| **与筛选并存** | 多选与范围选均在**当前网格 `assets`** 上工作；若开启标签筛选，该序列为子集——此时再执行「排序」写回，极易因成员数不一致而失败（见 12.15）。 |

### 12.8 右上角「⋯」操作菜单 — 图库 vs 相簿

菜单标题为 **「操作选项」**。下列各项中，**灰色**表示当前不可点（通常因**未选中任何照片**）；**撤销/重做**另受「栈里是否有可撤销/重做动作」控制。

| 菜单项 | 需先选中？ | 操作后典型结果 |
|--------|------------|----------------|
| 撤销 | 否（依赖栈） | 尝试撤销上一步受支持的相册类操作；无则可撤销项时灰掉。 |

|------|------|
| **标签筛选** | 通过搜索条相关入口打开 **「按标签筛选」** 面板；可选多个标签、切换「满足任一 / 满足全部」、重置。 |
| **结果** | 网格仅显示符合规则的照片；**不改变**用户在系统相簿中的真实成员表，只改变当前展示子集。 |
| **搜索栏输入** | **纯数字**：可解释为跳转到第 N 张（从 1 计数）并滚动到该位置；**文字**：按标签名子串匹配并应用筛选；**清空**：清除筛选状态。 |
| **条件** | 有激活筛选时，搜索区域宜保持可见以便用户感知当前处于过滤状态。 |

### 12.14 两种「粘贴」对比

| 方式 | 入口 | 插入位置语义 | 成功后常见表现 |
|------|------|----------------|----------------|
| **更多菜单 → 粘贴** | 右上角「⋯」 | **不按**某一格定位；将剪贴板资源加入当前相簿/集合（由系统与应用逻辑决定落点）。 | 刷新列表；可记录撤销。 |
| **长按格子 → 粘贴到此后方** | 长按缩略图上下文菜单 | **明确**插在长按的那一张**之后**。 | 列表立即更新；若此前非自定义排序常**自动切换**为自定义排序并保存偏好；可提示成功张数。 |

**条件**：剪贴板无照片资源时，菜单「粘贴」可能仍可点，但会提示失败；「粘贴到此后方」在剪贴板空时无效果或提示。

### 12.15 批量「排序」— 写回系统相簿顺序

| 项目 | 说明 |
|------|------|
| **位置** | 相簿详情 → 选择模式 → 选中至少两张 →「⋯」→ **排序**。 |
| **操作** | 建议：先理解**选中顺序**（点选/划选顺序），必要时长按设**锚点**；再点「排序」。过程中可出现「同步中」类进度提示。 |
| **结果（内存侧）** | 按 5.5 所述算法在**当前网格绑定的 `assets` 数组**上重排，得到**与该数组等长**的新序列；若此时 `assets` 为筛选子集，则后续系统校验通常会失败（见下行「标签筛选」）。 |
| **结果（系统侧）** | 将新顺序通过相册变更接口 **整表替换** 写回；成功后系统「照片」中打开该相簿应一致；成功时常**强制切到「按自定义排序」**并刷新 fetch，避免下次进入仍按日期拉取导致顺序丢失；并可记录撤销。 |
| **条件** | 至少 **两张** 选中；写回前服务层校验新序列与系统相簿成员 **数量相同、集合相同**。 |
| **失败** | 选中不足、锚点不在当前序列中、或与全量成员不一致等，应 **Alert**（如「至少需要选择两张照片…」「内部错误：无法找到作为排序基准的锚点照片」等）；**不应**长期呈现与系统不一致的「假成功」。 |
| **标签筛选** | 筛选开启时网格 `assets` 常为**真子集**，极易触发「数量不一致」校验失败；**建议先清除筛选再整簿排序**（见 5.5、12.7.1）。 |
